# Phase 6 Auth Deployment Gotchas — `use server` + Supabase Redirect URLs

**Captured:** 2026-05-29 (immediately after the incident)
**Status:** Three P0/P1-class production bugs shipped in PR #67, fixed in three follow-up hotfixes (PRs #68, #69, #70) within 4 hours of merge.
**Source:** E5.1 Phase 6 — `/app/verify-email`, `/app/forgot-password`, `/app/reset-password` (server actions + Supabase Auth integration).
**Audience:** Any developer or agent shipping Next.js 16 + Supabase Auth pages in this codebase. Also a cautionary note about how `npm run build` greenness can hide real production failures.

---

## TL;DR

Three independent gotchas combined to ship a broken auth slice to production:

1. **`'use server'` files in Next.js 16 may only export async functions.** Exporting any non-function const (e.g. `INITIAL_STATE = { ok: false }`) compiles cleanly locally but crashes at runtime in production with `A "use server" file can only export async functions, found object`. Fixed in PR #68.
2. **Supabase `auth.signUp` and `auth.resend` silently fall back to the dashboard "Site URL" when `emailRedirectTo` is omitted.** If Site URL is misconfigured (or defaults to `http://localhost:3000` from a prior dev session), every verification email points users at localhost. Fixed in PR #69.
3. **`NEXT_PUBLIC_SITE_URL` was not authoritative for the redirect-URL derivation.** Defence-in-depth: even with `emailRedirectTo` passed explicitly, if the env var isn't set the code fell back to dashboard-side resolution, which can drift. Fixed in PR #70.

Each individually is a 5-line code change. Combined, they took ~4 hours of incident response and three follow-up PRs.

## Gotcha 1: `'use server'` files can only export async functions

### Symptom

Production app crashes on first visit to any page that imports a server-action file:

```
A "use server" file can only export async functions, found object
```

`npm run build` and `npx tsc --noEmit` pass locally. Next.js dev server is forgiving; production runtime is not. The error surfaces only when the affected module is first loaded in production.

### Root cause

Each of the new auth actions files (`verify-email/actions.ts`, `forgot-password/actions.ts`, `reset-password/actions.ts`) opened with:

```ts
'use server';

import { /* … */ } from '…';

const INITIAL_STATE: VerifyEmailState = { ok: false };

export async function resendVerificationEmail(_state, formData) {
  // …
}
```

The pattern was copied from earlier Phase 5 actions, where it worked because those files were on Next.js 15. In Next.js 16, **anything exported (or even just declared at module scope and reachable from exports) from a `'use server'` file must be an async function**. The `INITIAL_STATE` const breaks the constraint.

Why local builds passed: Next.js 16's compile-time check evaluates the violation lazily; the failure only fires on first runtime evaluation of the module. Local `npm run build` produces the artefact without exercising it; the app only crashes when a real request reaches the page.

### Fix

Move `INITIAL_STATE` (and any other non-async-function exports) **out of the `'use server'` file** into a sibling client-side file (e.g. `verify-email/state.ts`) or inline them in the form component.

### How to catch this next time

- **Run the production build AND start the server** as a smoke test before merging: `npm run build && npm start`, then hit each affected page in a browser. Cheap, catches this class of failure in 30 seconds.
- **Lint rule** for this pattern would help. There isn't an ESLint plugin shipping it yet (as of Next 16.2.6) — worth raising upstream or writing a small custom rule.
- **Architectural rule**: keep `'use server'` files free of any module-scoped value other than the action functions themselves. Put types in a sibling `.types.ts` and constants in a sibling `.consts.ts`. The lint rule writes itself once that's enforced.

## Gotcha 2: Supabase `auth.signUp` silently falls back to dashboard Site URL

### Symptom

Production verification emails contain links like `http://localhost:3000/auth/callback?code=…` even though the app is hosted on `https://www.lslcalculator.com.au`. Users click the link, hit a connection refused, and can't complete signup.

### Root cause

```ts
const { data, error } = await supabase.auth.signUp({
  email,
  password,
  // emailRedirectTo not specified ← the bug
});
```

Supabase's `signUp` API ships an `options.emailRedirectTo` parameter that controls where the magic link points. When omitted, the SDK falls back to the project's dashboard **Site URL** setting. Site URL defaults to `http://localhost:3000` for new projects and is never automatically updated.

The Phase 6 actions never passed `emailRedirectTo`, trusting that "we configured the project properly". The project's Site URL had been set to `http://localhost:3000` during local-dev provisioning weeks earlier and never updated.

### Fix

Pass `emailRedirectTo` explicitly on every `auth.signUp`, `auth.resend`, `auth.resetPasswordForEmail`, and `auth.signInWithOtp` call:

```ts
const siteUrl = process.env.NEXT_PUBLIC_SITE_URL || 'https://www.lslcalculator.com.au';
const { data, error } = await supabase.auth.signUp({
  email,
  password,
  options: {
    emailRedirectTo: `${siteUrl}/app/verify-email`,
  },
});
```

### How to catch this next time

- **Integration test that asserts on the redirect URL** — generate a recovery link via the admin SDK and assert the link starts with the prod hostname, not localhost. Cheap, deterministic, runs in CI.
- **Pre-launch checklist item**: "Verify Supabase Dashboard → Auth → URL Configuration → Site URL matches production hostname." This is operational hygiene, not code, but should be in the launch guard.
- **Anti-pattern**: trusting "we configured it properly" for any setting that defaults to a dev value. Treat any setting whose default differs between dev and prod as a code-level concern, not a dashboard concern.

## Gotcha 3: `NEXT_PUBLIC_SITE_URL` wasn't authoritative

### Symptom

Even after Gotcha 2's fix, the code still allowed dashboard-Site-URL fallback in environments where `NEXT_PUBLIC_SITE_URL` wasn't set. The string literal fallback in Gotcha 2's fix was the right shape, but the actual implementation derived the redirect URL via a helper that could still hit Supabase's dashboard config under certain conditions.

### Root cause

Subtle: belts-and-braces, but with an inconsistent default chain. The fix in PR #69 used `process.env.NEXT_PUBLIC_SITE_URL` as the first source-of-truth but didn't enforce its presence — if it was unset, the code fell through to the Supabase SDK default, which in turn used the dashboard Site URL.

### Fix

Throw at module load if `NEXT_PUBLIC_SITE_URL` is unset in production:

```ts
const siteUrl = process.env.NEXT_PUBLIC_SITE_URL;
if (!siteUrl && process.env.VERCEL_ENV === 'production') {
  throw new Error('NEXT_PUBLIC_SITE_URL must be set in production');
}
```

This is fail-fast — the deploy goes red before users see broken emails.

### How to catch this next time

- **Fail-fast on missing env vars in production builds.** Cheap pattern; consistent with the spec's recommendation that the `Supabase environment variables are missing` error already follows. Apply the same pattern to every customer-facing URL env var.
- **Audit checklist**: for every env var that affects customer-facing URLs (Site URL, callback URLs, OAuth redirect URIs, magic-link templates), the code must assert presence and shape — never silently fall back.

## The meta lesson — green CI ≠ green production

All three bugs passed:
- `npx tsc --noEmit` (compile-time)
- `npx vitest run` (unit + integration with mocks)
- `npm run build` (build-time bundling)
- Playwright E2E (mocked / scoped)
- CI on PR #67 (the matrix that ran before merge)

They failed:
- Real user visiting the production app
- Real verification email sent from production Supabase

The class of failure is **module-load + runtime behaviour in the production environment**, which is invisible to every layer of pre-merge verification we have. Three concrete defenses:

1. **Always run a production smoke test before merging customer-facing surfaces.** `npm run build && npm start` against a localhost browser. The "is the app actually serving traffic?" check.
2. **Treat Vercel preview deploys as the real pre-merge check** for any auth-, payment-, or email-flow change. The preview deploy is a near-prod environment; click through every new page, trigger every new email path. If verification email shows `localhost`, that's the signal.
3. **For any "fall-back-to-dashboard-config" SDK pattern**, write a CI test that asserts the actual output (a real Supabase API call, a real email-content fetch). Mocking the SDK boundary masks exactly these failures.
