## Q1 — Migration 7 sequencing (LOAD-BEARING)

**Answer:** Order is safe **but the GIN index is currently double-specified.** Migration 7's table can land after Migration 2's column without any referential issue, because `employees.tags` (`text[]`) has no FK relationship to `tags.name` — Postgres can't FK an array element. The reference is enforced at the service layer, so column-then-table is fine. **However**, the plan asserts the GIN index lives in BOTH migrations and that is a real defect.

**Rationale.**
- Task 1.2 (Migration 2) explicit task body: *"**4 indexes** (`(org_id)`, `(org_id, archived_at)`, `(retention_expires_at) WHERE retention_expires_at IS NOT NULL`, **`USING GIN (tags)`**)"* — GIN here.
- Impl-plan §3.1 Migration 2 description: *"GIN index on `tags`: `CREATE INDEX employees_tags_gin_idx ON public.employees USING GIN (tags);`"* — GIN here too.
- Impl-plan §3.1 Migration 7 description: *"GIN index on `employees.tags` (added in this migration, references column from Migration 2)"* — GIN claimed here as well.
- Task 1.6b body: *"GIN index on `employees.tags` (added in this migration, references column from Migration 2)"* — and here.

The migration that actually writes `CREATE INDEX ... USING GIN (tags)` will succeed; the second one will fail with a duplicate-index error. **Recommendation:** put the GIN index in Migration 2 (it lives with the column it indexes, conventional, and Migration 2 is where the column is created). Remove the GIN-index claim from Migration 7. Migration 7 then carries only the `tags` dictionary table + cascade triggers + `usage_count_cached` maintenance trigger.

`DEFECT — needs PM amend.`

## Q5 — `bulkCreateFromImport` transaction boundary (LOAD-BEARING)

**Answer:** The cascade triggers themselves (rename / delete) are safe — they only fire on UPDATE / DELETE of `tags`, never on INSERT, so a bulk-create-from-import (insert-only) won't touch them. **But the `usage_count_cached` maintenance trigger is a real concern.** It fires on every `employees.tags` INSERT/UPDATE/DELETE, and for a 5k-employee onboarding import with an average of 2-3 tags per employee, that's 5k row-trigger invocations each doing N UPDATE-with-WHERE against the same handful of `tags` rows. Lock contention + transaction time blow up.

**Rationale.**
- `tg_cascade_tag_rename` — AFTER UPDATE OF name: fires only on `tags` rename. Not triggered by the bulk path.
- `tg_cascade_tag_delete` — BEFORE DELETE: fires only on `tags` row delete. Not triggered.
- `usage_count_cached` maintenance — the plan says *"one INSERT/UPDATE/DELETE trigger that compares OLD vs NEW tags arrays and adjusts cached counts."* For a 5k bulk insert this fires 5k times sequentially. Each firing does an UPDATE on `public.tags` (incrementing counters per distinct tag in the OLD/NEW diff). All those updates are against the same small set of dictionary rows. Postgres row-level locks within a single transaction are fine for correctness but kill throughput. Realistic prediction: a 5k-employee import on Supabase Pro Sydney would go from "few seconds" to "tens of seconds or worse," depending on tag distribution.

Three fixes (PM's call which to pick):
1. **Defer counter to end-of-import batch.** In the bulk path, set a session flag (or use a deferred trigger), do the bulk insert, then a single `UPDATE tags SET usage_count_cached = ... GROUP BY name` at the end. One round-trip per distinct tag instead of one per employee row.
2. **Drop `usage_count_cached` entirely.** Compute on demand for the org-settings tag-edit page via `SELECT name, cardinality(...)` or a view. Phase 4 only renders this page interactively; 0.5s extra render on settings is acceptable for the maintenance cost saved.
3. **Accept the cost.** v1 customer scale is ≤5k employees per IM-3, so a 30s import once at onboarding may be tolerable. Document it.

The trigger's existence is mentioned but its design is hand-waved (*"drafted at migration-write time"*). That's where the bug would land. **Surface the design decision before Migration 7 lands.**

`DEFECT — needs PM amend.` Recommend option 2 (drop the cache, compute on demand) — least code, no perf cliff, the tag-edit page is low-traffic.

## Q6 — Other stale forward-looking language

**Found two more PM missed:**

### 6a — Route group convention is stale

Impl-plan §5 + every Phase 4 task references routes under `/(authed)/` route group: `/(authed)/setup`, `/(authed)/employees`, `/(authed)/api/...`, `/(authed)/employees/[id]`, `/(authed)/employees/import`, `/(authed)/employees/new`, `/(authed)/settings/tags`.

**Actual codebase convention** (verified post-E5.1-Phase-6 — `ls website/src/app/`) uses a non-grouped `app/` segment: `website/src/app/app/login`, `website/src/app/app/logout`, `website/src/app/app/verify-email`, `website/src/app/app/forgot-password`, `website/src/app/app/reset-password`, `website/src/app/app/signup`. The deployment-gotchas doc itself says *"`/app/verify-email`, `/app/forgot-password`, `/app/reset-password`"*. No `(authed)` group exists.

A developer reading the impl-plan literally would create a parallel route hierarchy under `(authed)` which is non-conventional for this codebase and would yield URLs like `/setup` (route group is URL-transparent) — that's actually CORRECT from a URL perspective, but it diverges from the existing pattern where auth-required pages sit under `/app/<page>` instead. **The plan should say which convention it's following** and (preferably) match the existing E5.1 convention.

`DEFECT — needs PM amend.` Either confirm the codebase moves to a `(authed)` group (separate refactor in scope?), or update all Phase 4 route paths to `/app/setup`, `/app/employees`, `/app/employees/import`, etc.

RLS and trigger tests live here. **L (12–16 hrs)**."* — but the rest of the file (§9 effort summary, Task 1.10) says **seven** migrations and **14–18 hrs**. Stale leftover from before the rebase amend. Same drift inside impl-plan §2 Phase breakdown table: row says *"Phase 1 ... L (12–16 hrs)"* while §9 says L+ 14-18.

`DEFECT — needs PM amend.` Trivial doc fix; not load-bearing but visible inconsistency.
