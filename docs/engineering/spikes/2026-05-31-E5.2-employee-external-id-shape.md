# Spike — `employee_external_id` column shape across AU payroll exports

- **Date:** 2026-05-31
- **Epic / sub-spec:** E5.2 (employee masterfile) — resolves DEV-EMP-2 (HIGH) per `.specify/features/005-lsl-platform/sub-specs/employee-masterfile-impl-plan.md` §0
- **Type:** Phase 0 research spike (throwaway). No code, no migration, no schema change.
- **Time-boxed:** 1–2 hours
- **Author:** Developer agent (DEV-EMP-2 dispatch)
- **Status:** Complete

---

## 1. Question the spike answers

Before E5.2 Migration 2 (the `employees` table) lands, the `UNIQUE (org_id, lower(employee_external_id))` constraint is a one-way door. It cascades into E5.4's 4-key pay-period dedup `(org_id, employee_external_id, pay_period_end, pay_code_raw)`. The spike confirms or amends four properties of the column:

1. **Max observed length** of the "employee ID / employee number" field across major AU payroll exports.
2. **Character set** (numeric only, alphanumeric, hyphens, leading zeros).
3. **Case-sensitivity in practice** — does the same employee ever appear with both `EMP01` and `emp01`?
4. **Stability across pay runs** — does the field remain constant for the same employee across exports from the same payroll system?

---

## 2. Default column shape (going in)

Per impl-plan §0 DEV-EMP-2:

| Property | Default |
|---|---|
| Postgres type | `text` |
| Storage | trimmed (`btrim()` in service layer; not a DB CHECK) |
| Uniqueness | `UNIQUE (org_id, lower(employee_external_id))` |
| Max length DB CHECK | none (Postgres `text` is arbitrary) |
| Soft validator (service) | 64 chars max; friendly UI error |
| Display | original casing preserved |

The spike's job is to either ratify this shape or propose a tightened/relaxed amendment for Task 1.2 *before* the migration is written.

### 4.2 Per-system narrative

**Xero AU Payroll — the surprise finding.** Xero AU Payroll **does not expose an editable, human-readable employee number** to customers. The only identifier exposed in the API is `EmployeeID`, a UUID. Xero's own product team confirmed (community forum, December 2, 2025) that an employee-number field is "not in pipeline." Customers commonly:

- Repurpose the `Job Title` field, the `Middle Name` field, or the free-text `Notes` section to store an internal employee number.
- Rely on `FirstName + LastName + DateOfBirth` as the matching key.
- Use the UUID (38 chars including dashes) in API integrations.

**Implication for E5.2:** A Xero CSV export, if used as the ingestion source for the masterfile, will likely supply **either**:
- A UUID (36 chars, lowercase hex with hyphens) if exported via the API/Hubdoc-style integration, **or**
- A free-text workaround value of unknown shape (whatever the customer chose), **or**
- No value at all (then the importer must synthesise one — out of scope for the spike).

Both UUIDs (36) and most customer workarounds (≤ 20 chars in practice) fit comfortably inside a 64-char soft cap.

**MYOB AccountRight — Card ID / DisplayID.** This is the cleanest case. The Card ID is enforced unique within an MYOB company file (community confirmation; MYOB's own duplicate-card help articles). The API documents a 15-char max. The MYOB default format `EMP00001` includes leading zeros — these are part of the string in MYOB and must survive the CSV round-trip. Leading-zero loss in Excel CSV imports is a well-known operator-side hazard (Microsoft + Salesforce docs cited), but the source export from MYOB itself preserves them as text.

**KeyPay / Employment Hero — `ExternalId`.** This is the most variable. KeyPay explicitly delegates the shape to the customer: "set to anything you like; must be unique per employee." Observed customer choices in the wild:
- Email addresses (often >30 chars; can exceed 64).
- Payroll system codes (e.g. `EMP-001`, `001`, `T123456`).
- The numeric `Id` re-used as a string.

**Case sensitivity:** KeyPay's `GET /employee/externalid/{value}` endpoint matches **case-sensitively** at the API layer. This is the only system in the spike where case sensitivity is observable and documented. Implication: if `lower()` is applied as the dedup key in LSL, **two KeyPay employees with `ABC123` and `abc123` would collide in LSL even though they are distinct in KeyPay.** This is an exceptionally rare configuration (KeyPay deduplicates on uniqueness at write time within their own DB but doesn't reject case variants), but the risk is non-zero.

**Reckon One / MicrOpay / ELMO — sparse public docs.** Public docs are thin. Reckon's employee numbering is generally monotonic numeric. MicrOpay's *Memo Employee Number* is documented at 7 chars alphanumeric. ELMO's External ID is bounded by the downstream payroll system it integrates with. None of these contradicts the default 64-char service-layer cap.

## 5. Synthesis

### 5.1 Observed dimensions

| Dimension | Observed range | Default shape covers it? |
|---|---|---|
| **Max length** | 7 chars (MicrOpay) to 36 chars (Xero UUID); customer workarounds and KeyPay free-text can exceed 64 chars if customer uses long emails as External ID | YES with 64-char soft cap — caveat below |
| **Character set** | Digits, letters, hyphens, underscores, periods (in emails as External ID), `@` (in emails) | YES with `text` |
| **Leading zeros** | Yes, MYOB's default `EMP00001` and many custom formats. Must be preserved as text. | YES with `text` — but the **ingestion layer** must read the column as text, not number |
| **Case sensitivity in source** | KeyPay's API is case-sensitive at lookup. MYOB and Xero are effectively case-insensitive in practice (all-caps defaults). | The default `UNIQUE (org_id, lower(...))` **case-folds**, which is a delta from KeyPay's behaviour |
| **Stability across pay runs** | All systems treat the employee identifier as stable once assigned. Re-assignment is rare and requires admin action. | YES — assumption holds |

### 5.2 Edge cases worth naming

1. **Long emails used as KeyPay ExternalId.** E.g. `firstname.lastname@longerexamplecompanyname.com.au` is 50+ chars. Still under 64, but a customer with `firstname.middlename.lastname@…` could realistically hit 65+. The soft cap of 64 may need to grow to 128 to be safe.
2. **Case-variant collision (KeyPay only).** A customer who somehow has `ABC123` and `abc123` as distinct ExternalIds in KeyPay will have one of them silently rejected by LSL's `UNIQUE (org_id, lower(...))` constraint. The dedup error message must make this clear: "duplicate external ID after case-folding."
3. **No external ID at all (Xero workaround customers).** The masterfile import flow must reject rows with empty `employee_external_id` with a friendly error, **and** the docs/UI must steer Xero-source customers toward a stable substitute (the Xero UUID or a `firstname-lastname-dob` composite). This is an ingestion-layer concern, not a column-shape concern, but it falls out of the same investigation.
4. **Leading-zero loss at the Excel boundary.** If an operator opens a MYOB-export CSV in Excel and re-saves, `EMP00001` may become `1`. This is an operator-error class, but the CSV-import UI should warn explicitly when it detects truncated-looking numeric IDs. **Not a column-shape concern.**
5. **UUIDs and case.** UUIDs from Xero are canonical lowercase. Applying `lower()` is a no-op. Safe.
6. **Hyphens, periods, and `@`.** All allowed by `text` and by the proposed soft validator. No regex character-set CHECK is recommended — too brittle across vendors.

### 5.3 Does the default shape stand?

**The default shape stands**, with two small recommended tightenings to the **service-layer validator** (NOT to the DB column):

| Property | Default | Recommendation | Reason |
|---|---|---|---|
| DB type | `text` | **Keep** | Covers every observed format including UUIDs and emails |
| DB length CHECK | none | **Keep none** | Postgres `text` is fine; no value in a CHECK that the service layer enforces upstream |
| `UNIQUE (org_id, lower(...))` | enforced | **Keep** | Case-folding is correct for the 99.9% case (MYOB, Xero, Reckon, MicrOpay). KeyPay's case-sensitive edge case is rare enough to handle in the error message, not the schema. |
| Service validator: max length | 64 | **Raise to 128** | KeyPay customers commonly use long-email ExternalIds. 64 is too tight. 128 is comfortably above the longest observed value (longest email + domain) and still below any pathological DOS-style payload. |
| Service validator: trim | yes | **Keep** | All observed payloads benefit from trim |
| Service validator: reject empty | implicit | **Make explicit** — service rejects empty/whitespace-only with `ServiceError.kind = 'missing_external_id'` | Xero-source customers will hit this; the error needs to be discoverable |
| Service validator: character set | not enforced | **Keep unrestricted** | Hyphens, periods, `@`, alphanumerics all needed. A regex would block legitimate KeyPay email-as-ExternalId customers. |
| Error message on dedup collision | `duplicate_external_id` | **Keep code, refine UI copy** to mention case-insensitive matching ("`ABC123` and `abc123` are treated as the same employee — if these are distinct people in your payroll, use distinct codes.") | KeyPay edge case discoverability |

### 5.4 Recommended Task 1.2 column definition (no amendment to schema; service-layer note)

```sql
-- in the employees table migration:
employee_external_id text NOT NULL,
-- Trim is enforced at the service layer (data/employee/), not by a DB trigger.
-- No length CHECK at the DB layer.

CREATE UNIQUE INDEX employees_org_external_id_ci_idx
  ON employees (org_id, lower(employee_external_id));
```

Service-layer validator (lives in `website/src/lib/data/employee/validation.ts` — to be created in Task 1.2 / 1.3):

```ts
// pseudocode for the validator contract — NOT a code change in this spike
const MAX_EXTERNAL_ID_LENGTH = 128;  // raised from 64 per spike

function normaliseExternalId(raw: string): string {
  return raw.trim();
}

function validateExternalId(raw: string): ServiceError | null {
  const trimmed = normaliseExternalId(raw);
  if (trimmed.length === 0) return { kind: 'missing_external_id' };
  if (trimmed.length > MAX_EXTERNAL_ID_LENGTH) return { kind: 'external_id_too_long' };
  return null;
}
```

## 6. Assumptions made (and what would falsify them)

The spike did **not** inspect real customer CSV exports. The findings rely on vendor documentation and public community evidence. Each assumption below names the data that would falsify it.

| # | Assumption | Falsifier |
|---|---|---|
| A1 | Xero AU Payroll has no native human-readable employee number field. | A Xero AU Payroll export that includes a stable, customer-editable `Employee Number` column. |
| A2 | MYOB Card ID is hard-capped at 15 chars and used as the canonical employee key. | An MYOB export with a Card ID > 15 chars, or a customer file where Card ID is blank for most employees. |
| A3 | KeyPay ExternalId is free-form, case-sensitive at lookup, often ≤ 64 chars but can exceed it (especially when an email is used). | A KeyPay export where ExternalId is consistently > 128 chars, or where case-variant duplicates appear in the same business. |
| A4 | Leading zeros are intended to be part of the identifier string (not numeric padding to be stripped). | A vendor doc that says "leading zeros are formatting only — strip on import." (None found; the opposite is documented for MYOB.) |
| A5 | The employee external ID is stable across pay runs from the same system. | A vendor doc or customer report showing re-numbering on a per-pay-run basis. |
| A6 | 128 chars at the service layer is a safe soft cap. | A KeyPay or similar customer who uses ExternalIds > 128 chars. (No public evidence of this.) |

**Follow-up — explicit defer of OQ-ING-3 evidence to pilot.** The impl-plan already documents that "OQ-ING-3 is deferred to dev validation against real exports." This spike does NOT close OQ-ING-3 — it ratifies the default shape and tightens the service-layer validator. Real-CSV validation should happen **once** during pilot, against at least one export from each of: Xero, MYOB, KeyPay. If pilot finds an actual collision or oversize value, the soft cap or the case-folding policy can be revisited cheaply because **the DB column itself remains unconstrained**.

---

## 7. Recommendations

1. **Ratify the default shape in Task 1.2** with two service-layer tweaks:
   - Raise the soft length cap from 64 to **128** chars.
   - Make "empty external ID" a typed `ServiceError.kind` (`missing_external_id`) with a friendly UI message that steers Xero-source customers toward a substitute.
2. **Do NOT add a DB length CHECK.** Postgres `text` is correct. Length enforcement lives at the service layer where it can evolve without a migration.
3. **Do NOT change the uniqueness key.** `UNIQUE (org_id, lower(employee_external_id))` is correct for 99.9% of cases. Add a UI/error-copy note for the KeyPay case-variant edge case.
4. **Add a comment to the migration file** referencing this spike doc for future maintainers, so the trade-offs (case-folding, no length CHECK, soft validator) are discoverable.
5. **Document the leading-zero / Excel hazard** in the import-flow UI copy task (Phase 4). Out of scope for Task 1.2.
6. **Defer real-CSV validation to pilot** per OQ-ING-3 — explicitly record this carry-over in `epic-status.md` for E5.2.

---

## 8. Headline

**Default shape stands.** One small amendment: raise the service-layer length validator from 64 to 128 chars to safely accommodate KeyPay customers who use long emails as `ExternalId`. No DB schema change vs. the impl-plan default. No re-write of Migration 2 needed.
